# Frontend Design Guide Checklist

토스 프론트엔드 가이드라인 기반 O/X 체크리스트입니다.
커밋 전 스테이징된 `.ts` / `.tsx` 파일이 각 항목을 만족하는지 검사합니다.

---

## 1. Readability (가독성)

### 1-1. 매직 넘버 명명
- [ ] **R-01** 의미 있는 숫자 리터럴(지연 시간, 임계값, 재시도 횟수 등)이 명명된 상수로 추출되어 있는가?
  - 예외: `0`, `1`, `-1`, 배열 인덱스, 명백한 수학 상수(`Math.PI`)
  - 나쁜 예: `await delay(300)`
  - 좋은 예: `const ANIMATION_DELAY_MS = 300; await delay(ANIMATION_DELAY_MS);`

### 1-2. 구현 세부사항 추상화
- [ ] **R-02** 인증 체크, 권한 검사 등 공통 로직이 컴포넌트 본문에 직접 작성되지 않고 래퍼/가드 컴포넌트로 분리되어 있는가?
- [ ] **R-03** 다이얼로그/오버레이 등 복잡한 인터랙션이 전용 컴포넌트로 추출되어 있는가?

### 1-3. 조건부 렌더링 분리
- [ ] **R-04** 역할(role)이나 상태에 따라 크게 다른 UI/로직이 하나의 컴포넌트 안에서 긴 삼항/조건문으로 처리되지 않고, 별도 컴포넌트로 분리되어 있는가?

### 1-4. 복잡한 삼항 연산자 단순화
- [ ] **R-05** 중첩 삼항 연산자(`a ? b : c ? d : e`)가 `if/else` 또는 IIFE(즉시 실행 함수)로 대체되어 있는가?

### 1-5. 시선 이동 감소 (Colocation)
- [ ] **R-06** 단일 사용처에서만 쓰이는 단순 로직이 불필요하게 상단/외부로 추출되지 않고, 사용되는 위치 근처에 배치되어 있는가?

### 1-6. 복잡한 조건에 이름 붙이기
- [ ] **R-07** 2개 이상의 조건이 조합된 boolean 표현식이 의미 있는 변수명으로 추출되어 있는가?
  - 나쁜 예: `if (product.categories.some(c => c.id === targetId) && product.prices.some(p => p >= min && p <= max))`
  - 좋은 예: `const isSameCategory = ...; const isPriceInRange = ...; if (isSameCategory && isPriceInRange)`

## 2. Predictability (예측 가능성)

### 2-1. 반환 타입 표준화
- [ ] **P-01** 유사한 목적의 함수/훅(예: API 훅들)이 일관된 반환 타입 구조를 사용하는가?
  - React Query 훅: 모두 `UseQueryResult` 객체 반환 (데이터만 구조 분해해서 반환 X)
- [ ] **P-02** 검증(validation) 함수들이 일관된 타입(예: Discriminated Union `{ ok: true } | { ok: false; reason: string }`)을 반환하는가?

### 2-2. 숨겨진 사이드 이펙트 제거 (SRP)
- [ ] **P-03** 함수가 이름에 드러나지 않은 사이드 이펙트(로깅, 분석 이벤트, 전역 상태 변경 등)를 수행하지 않는가?
  - 나쁜 예: `fetchBalance()` 함수 내부에서 `logging.log('balance_fetched')` 호출
  - 좋은 예: 호출부에서 명시적으로 로깅

### 2-3. 고유하고 설명적인 네이밍
- [ ] **P-04** 표준 라이브러리를 래핑한 함수/모듈이 원본과 구분되는 고유한 이름을 가지는가?
  - 나쁜 예: 인증 토큰을 자동 주입하는 `http.get` (내장 http와 동일한 이름)
  - 좋은 예: `httpService.getWithAuth`

## 3. Cohesion (응집도)

### 3-1. 폼 응집도 선택
- [ ] **C-01** 폼 검증이 용도에 맞게 필드 단위(독립 검증) 또는 폼 단위(통합 스키마)로 일관되게 선택되어 있는가?
  - 필드 단위: 독립적 검증, 비동기 검증, 재사용 필드
  - 폼 단위: 상호의존 필드, 위저드 폼, 단일 스키마

### 3-2. 도메인별 디렉토리 구조
- [ ] **C-02** 기능/도메인 관련 코드가 `components/`, `hooks/`, `utils/` 로 타입별 분산되지 않고 도메인 폴더(`domains/user/`, `features/post/` 등)에 묶여 있는가?
  - 공통 유틸/UI는 최상위 `components/`, `hooks/`, `utils/`에 둠 (허용)

### 3-3. 상수와 로직의 근접성
- [ ] **C-03** 상수가 해당 상수를 사용하는 로직과 같은 파일 또는 가까운 위치에 정의되어 있는가?
  - 전역 상수 디렉토리에 있더라도 이름으로 용도가 명확한가?

---

## 4. Coupling (결합도)

### 4-1. 성급한 추상화 지양
- [ ] **CP-01** 단순히 코드가 비슷하다는 이유로 섣불리 공통 훅/함수로 추출되지 않았는가?
  - 사용처별로 요구사항이 분기할 가능성이 있다면 중복을 허용
  - 판단 기준: "이 로직은 앞으로도 모든 사용처에서 동일하게 유지될 것인가?"

### 4-2. 상태 관리 범위 축소
- [ ] **CP-02** 여러 관심사(예: cardId, dateRange, filter)가 하나의 훅/컨텍스트에 묶이지 않고, 관심사별로 분리된 훅으로 나뉘어 있는가?

### 4-3. Props Drilling 제거
- [ ] **CP-03** 3단계 이상 props가 그대로 전달되는 경우 컴포넌트 컴포지션(`children`)으로 대체되어 있는가?
  - 중간 컴포넌트가 props를 사용하지 않고 하위로만 전달한다면 컴포지션으로 리팩터링

---

## 검사 규칙

- 각 항목은 스테이징된 파일에 **해당 패턴이 존재할 때**만 검사합니다 (예: 폼 관련 항목은 폼이 있을 때만).
- 해당 패턴이 파일에 존재하지 않으면 `N/A`(검사 제외)로 처리합니다.
- `X` 판정 시 파일 경로와 라인 번호, 위반 내용, 수정 제안을 함께 출력합니다.
- 모든 항목이 `O` 또는 `N/A` 일 때만 커밋이 승인됩니다.
